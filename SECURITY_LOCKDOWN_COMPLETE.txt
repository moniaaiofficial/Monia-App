Policy: select_own_chats
  Operation: SELECT
  Condition: auth.uid()::text = ANY(participants)
  Effect: User sees only chats where they are participant
  Security: ✓ Data isolation

Policy: insert_own_chat
  Operation: INSERT
  Condition: auth.uid()::text = ANY(participants)
  Effect: Creator must be in initial participants array
  Security: ✓ Prevents unauthorized chat creation

Policy: block_chat_updates
  Operation: UPDATE
  Condition: USING (false)
  Effect: NO updates allowed to any chat
  Security: ✓ Participants are immutable

Policy: block_chat_delete
  Operation: DELETE
  Condition: USING (false)
  Effect: NO deletes allowed
  Security: ✓ Data preservation

Policy: select_messages
  Operation: SELECT
  Condition: EXISTS (SELECT FROM chats WHERE...)
  Effect: Messages visible only from participant chats
  Security: ✓ Cross-chat isolation

Policy: insert_own_message
  Operation: INSERT
  Condition: auth.uid()::text = sender_id
  Effect: Can only send as yourself
  Security: ✓ Prevents impersonation

Policy: update_own_message

  Effect: Can only delete your own messages

AUTHENTICATION & AUTHORIZATION
════════════════════════════════════════════════════════════════════════════════

Authentication Method: JWT (Clerk)
├─ User logs in with Clerk
├─ Clerk issues JWT token
├─ Client includes JWT in requests
└─ Supabase extracts auth.uid() from JWT

Authorization Method: RLS Policies
├─ Every operation evaluated against JWT
├─ auth.uid()::text compared to policy conditions
├─ Policy returns TRUE = operation allowed
└─ Policy returns FALSE = operation rejected

Data Isolation
├─ Each user sees only their data
├─ Cross-user access: IMPOSSIBLE
├─ Chats filtered by participants: [user_id, ...]
└─ Messages filtered by chat membership
